# บทนำ

## บทนี้จะได้เรียนรู้อะไร

เมื่อจบบทนี้ ผู้เรียนจะอธิบายวงจรแจ้งซ่อมตั้งแต่ผู้ใช้งานส่ง Ticket จน Supervisor ตรวจรับงานได้ แยกบทบาทของ Power Apps, Power Automate, Supabase และ Power BI ได้ และเลือกจุดเริ่มต้นของโครงการให้เหมาะกับบริบทองค์กร

## ปัญหาที่ต้องการแก้

งานซ่อมบำรุงที่กระจายอยู่ในแชต, โทรศัพท์ และ Excel ทำให้ติดตาม SLA ยาก ไม่รู้ว่าใครรับผิดชอบ และค้นหาประวัติ Asset ไม่ได้ CMMS จึงต้องทำหน้าที่เป็นแหล่งข้อมูลกลาง (system of record) พร้อมเก็บหลักฐานการทำงาน

## แนวคิดพื้นฐาน

CMMS ประกอบด้วย 5 ส่วน: master data ของ Asset, การรับแจ้งปัญหา, Work Order, หลักฐานการซ่อม และการวิเคราะห์ผล แต่ไม่ควรเริ่มจากหน้าจออย่างเดียว ต้องเริ่มจาก workflow และกฎธุรกิจ เช่น งาน Critical ต้องแจ้ง Supervisor และไม่อนุญาตให้ปิดงานก่อนมีผลการตรวจรับ

## ผลลัพธ์ของ Final Project

ระบบตัวอย่างจะรองรับสถานะ `New → Submitted → Assigned → In Progress → Completed → Verified → Closed` พร้อมเก็บประวัติการเปลี่ยนสถานะ และให้ผู้ใช้ค้นหาจาก Ticket, Asset, Site หรือช่วงวันที่

## วิธีใช้คู่มือนี้

1. อ่านแนวคิดสั้น ๆ แล้วทำ Lab ให้ครบ
2. คัดลอกโค้ดไปทดลองใน environment แยกจาก Production
3. บันทึกผลลัพธ์และปัญหาที่พบในสมุดงานของตนเอง
4. ทำ Mini Project ทุก Phase เพื่อให้ artefact ต่อเนื่องถึง Final Project

> **Warning:** ตัวอย่างนี้ใช้ข้อมูลจำลอง ห้ามนำชื่อผู้ใช้งานจริง, รูปหน้างานจริง หรือ key ของ Supabase มา commit ใน repository
